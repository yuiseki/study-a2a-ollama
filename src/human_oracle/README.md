# human_oracle

A2A メッシュに「人間そのもの」を 1 ノードとして参加させる実験。

`TASK_STATE_INPUT_REQUIRED` で停止するのではなく、独立した Agent Card を持つ **"Human Oracle"** として人間を露出し、他のエージェントから普通に `SendMessage` で呼び出せるようにする。

## なぜ "agent" ではなく "oracle" か

これは A2A プロトコル仕様が定義した用語ではなく、このデモでの呼称選択。**「自力では答えに辿り着けないシステムが、外部にある真実の源に問い合わせる」** という意味を持たせたかったため。

### 語源と系譜

| 文脈 | oracle の意味 |
| --- | --- |
| 古代ギリシャ（デルフォイ） | 神託を告げる巫女。人間には知り得ない情報の源 |
| 計算理論（Turing, 1939 頃） | 決定的な計算機が解けない問いに答えを与える黒箱（oracle machine） |
| 暗号理論 | プロトコルの証明で使う理想化された外部関数（random oracle model 等） |
| Ethereum / スマートコントラクト | チェーン内では取れない外部世界の真実を橋渡しする存在（Chainlink 等） |

どれも共通して「**決定論的なシステムから見て、自分の外部にある真実の源**」を指す。

## A2A 文脈での Oracle

LLM エージェントが**原理的に答えられない問い**が存在する。訓練データの中に答えが無く、推論でも導けず、他 AI エージェントに委譲しても解決しないもの。こうした問いに対して「人間」が唯一の情報源として機能する。この構造が Ethereum の Oracle Problem（スマートコントラクトが外部価格を知る問題）と相似的で、A2A メッシュでは **Human Oracle** という形で同じ解を提供できる。

### AI が原理的に持てない情報の類型

| 種類 | 例 | なぜ AI では無理か |
| --- | --- | --- |
| 主観的感覚 | 「今日の気温はどう感じますか」「この匂いは好き？」 | クオリアそのもの。LLM の訓練データは他人の主観の集計で、本人のそれではない |
| 個人の現在状態 | 「今どんな気分？」「体調は？」 | モデルの中に本人のリアルタイム状態は無い |
| 美的判断 | 「この街の景色は魅力的か？」「このロゴ案は良いか？」 | 個人の好みが収束しない。平均化すると凡庸になる |
| 未公開知識 | 「この契約の裏合意は何か？」 | 訓練データに無い |
| 規範的判断 | 「これは倫理的に OK か？」 | 文脈ごとに正解が揺れ、責任の所在が AI には持てない |

これらはすべて **Human Oracle が存在理由を持つ領域**。逆に「3+5 は？」のような決定論的問いを Human Oracle に投げるのはナンセンス。このデモで敢えてそれをやっているのは仕組みの確認が目的。

## Agent Card の skills 設計パターン

本デモでは単一の `ask_human` skill にまとめているが、実用時は **何を oracle として保証するか**でスキルを分割した方が発見・ルーティングがしやすい。

```python
# 例: 主観的判断用の Human Oracle
AgentSkill(
    id="subjective_judgment",
    name="主観的判断",
    description="嗜好・美的・感覚的な問いに、この人間個人の視点で答える",
    examples=["このロゴ案 A/B どっちが良い？", "この BGM は会議室の雰囲気に合う？"],
)

# 例: 承認権限を持つ Human Oracle
AgentSkill(
    id="policy_approval",
    name="ポリシー承認",
    description="組織ポリシー観点の判断と責任署名",
    examples=["この広告文言は社外公開してよいか", "この契約条件は承認できるか"],
)

# 例: 倫理判断の Human Oracle
AgentSkill(
    id="ethical_review",
    name="倫理レビュー",
    description="AI が下すべきでない価値判断を、責任を負える人間が行う",
    examples=["この機能はユーザーを誤誘導していないか", "この個人情報の取り扱いは適切か"],
)
```

ポイントは **skill の description に「人間にしか答えられない理由」が書かれていること**。これがあると上流のエージェントは `ask_human` を使うか自分で答えるかを skills カタログから選べる。

## Oracle Problem と A2A

Ethereum 界隈で議論される "Oracle Problem" は A2A にも同型で現れる。

| 観点 | Ethereum oracle | A2A human oracle |
| --- | --- | --- |
| 単一オラクルは信頼できない | 1 つの価格情報源が嘘をつくかも | 1 人の判断は偏るかも |
| 解決アプローチ | Chainlink 等の分散オラクル網で多数決 | 複数人にブロードキャストして合意を集約するメタ Oracle |
| 真偽の保証 | 署名 + staking + reputation | A2A v1.0 の **signed Agent Card**（JWS + RFC 8785）+ reputation |
| フリーライド防止 | 経済的インセンティブ設計 | 組織の役割権限、監査ログ |

A2A v1.0 の署名付き Agent Card は「このノードは本物か」を検証する下地になっており、Human Oracle の認証にも使える。これを土台に「**複数の Human Oracle を束ねてコンセンサスを返すメタ Oracle**」を構築することも理論上可能。人間の主観判断を分散システムの 1 次情報として扱うインフラとして A2A が機能し始めると、単なるエージェント間 RPC の枠を超えて、**人間の集合意思決定までをプロトコル化できる**領域に入る。

## このデモの限界

- Human Oracle の応答は **stdin 依存**。本番では Slack / メール / Web UI を front に置く
- 応答時間の契約が無い（SLA が無いので Task は永遠に `working` に留まれる）
- **認証・認可が無い**。誰でも Human Oracle を叩けてしまう。実用化するなら Bearer token / mTLS + 承認者プール
- 単一ノードなので Oracle Problem 的リスクが残る

これらは意図的にスコープ外にしている。プロトコル上の概念検証のみを目的とするミニマム実装。
